1. Vue d'ensemble

1.1 Objectif

Le Manual Trend Extractor est une application web interne Siemens développée en ASP.NET Core (Razor Pages) permettant à un opérateur de déclencher manuellement l'extraction de données de tendances (trends) depuis un serveur Desigo CC via l'API NORIS RESTful v8.0.

Contrairement au Trends Extractor automatique (v4.0), cet outil laisse le contrôle total à l'utilisateur : il choisit la plage temporelle souhaitée, lance l'acquisition à la demande et suit la progression en temps réel depuis son navigateur web. L'outil est conçu pour des besoins ponctuels, des re-extractions ou des vérifications manuelles de données historiques.

⚠️ Statut : Cette application est actuellement en version v.1 et n'a pas encore été déployée en environnement de production. Elle est en cours de développement et de validation.

4. Modes d'Acquisition

4.1 Mode Date Unique

Le mode Date unique permet d'extraire l'ensemble des données de tendances pour une journée complète. La plage horaire est automatiquement fixée à :

  • Début : 00:00:00 de la date sélectionnée
  • Fin : 23:59:59 de la date sélectionnée
ℹ️ Valeur par défaut : La date est initialisée à la date du jour au chargement de la page via @DateTime.Today.ToString("yyyy-MM-dd") côté serveur Razor.

Cas d'usage typiques

  • Vérification des données d'une journée spécifique
  • Re-extraction d'une journée après une panne
  • Audit ponctuel d'une date précise

4.2 Mode Période

Le mode Période permet de définir une plage de dates personnalisée avec une date de début (Du) et une date de fin (Au).

Champ Valeur par défaut Description
Du Date du jour − 7 jours Date de début de la période d'extraction
Au Date du jour Date de fin de la période d'extraction
⚠️ Attention aux grandes périodes : Plus la période est longue, plus le volume de données est important. Des périodes très étendues peuvent :
  • Générer des timeouts API (délai de 180s par requête)
  • Augmenter significativement le temps de traitement
  • Produire des fichiers de sortie volumineux
Il est recommandé de ne pas dépasser 30 jours par extraction.

Cas d'usage typiques

  • Re-extraction d'une semaine après une interruption du service automatique
  • Extraction pour un rapport mensuel
  • Récupération de données historiques pour analyse

4.3 Annulation d'une Acquisition

Il est possible d'interrompre une acquisition en cours à tout moment via le bouton ⛔ Arrêter qui apparaît automatiquement pendant le traitement.

Comportement Description
Déclenchement Clic sur le bouton ⛔ Arrêter → appel de cancelAcquisition()
Effet immédiat L'acquisition s'arrête proprement après le trend en cours
Données partielles Les trends déjà traités avant l'annulation sont conservés
Retour UI Le bouton Arrêter disparaît, le bouton Lancer redevient actif
ℹ️ Visibilité du bouton : Le bouton ⛔ Arrêter possède la classe si-hidden par défaut. Il devient visible uniquement lorsqu'une acquisition est en cours, géré dynamiquement par app.js.

5. Gestion de Session et Token

5.1 Authentification OAuth 2.0

L'application utilise le protocole OAuth 2.0 avec un token Bearer pour authentifier toutes les requêtes vers l'API NORIS de Desigo CC. Le token est obtenu lors de la connexion initiale et doit être renouvelé avant son expiration.

Durées de validité du token

Valeur expires_in Durée Type de configuration Comportement TokenMonitor
1500 25 minutes Configuration courte Alerte rapide, refresh fréquent recommandé
86400 24 heures Configuration standard Surveillance normale
2591999 ~30 jours Configuration longue durée Surveillance passive (ex: log affiché dans la console)
ℹ️ Contrôle serveur : La durée de validité du token est entièrement définie par le serveur Desigo CC. L'application cliente ne peut pas modifier cette valeur. Elle est configurée par l'administrateur système côté Desigo CC.

5.2 TokenMonitor

Le TokenMonitor est un script JavaScript (tokenMonitor.js) qui surveille en continu la validité du token OAuth et alerte l'utilisateur avant son expiration.

Comportement du TokenMonitor

Événement Action Message dans les logs
Démarrage de la surveillance Initialisation du timer interne 🕐 [TokenMonitor] Surveillance démarrée (expire dans Xs)
Token proche de l'expiration Mise à jour de l'indicateur Token en orange Avertissement dans la console
Token expiré Indicateur Token en rouge, acquisition bloquée Message d'erreur dans la console
Token renouvelé Réinitialisation du timer, indicateur vert Confirmation dans la console
⚠️ Acquisition et expiration : Si le token expire pendant une acquisition en cours, celle-ci risque d'échouer pour les trends non encore traités. Il est recommandé de vérifier l'état du token avant de lancer une extraction longue.

5.3 Refresh Token Manuel

Le bouton 🔄 Refresh Token dans le header permet de renouveler manuellement le token à tout moment, sans avoir à recharger la page.

Flux d'exécution du Refresh Token

  1. Clic sur le bouton → appel de la fonction refreshToken()
  2. Désactivation du bouton → classe CSS loading appliquée (opacity réduite, curseur attente)
  3. Requête POST vers /api/token avec les credentials configurés
  4. Réception du nouveau token → mise à jour du token en mémoire
  5. Réinitialisation du TokenMonitor → nouveau timer de surveillance
  6. Réactivation du bouton → classe loading retirée
  7. Log de confirmation → message affiché dans la console
✅ Bonne pratique : Effectuer un refresh token avant de lancer une extraction longue (période de plusieurs jours) pour s'assurer que le token ne va pas expirer pendant le traitement.

5.4 Connexion SignalR

L'application utilise SignalR pour recevoir les mises à jour en temps réel depuis le serveur (logs, progression, résultats) sans avoir à rafraîchir la page.

Aspect Détail
Librairie js/signalr.min.js (hébergée localement)
Établissement Automatique au chargement de la page
Reconnexion Automatique en cas de perte de connexion
Messages reçus Logs, progression, résultats d'acquisition
Log de confirmation 🔗 Connexion SignalR établie.
⚠️ Perte de connexion SignalR : Si la connexion SignalR est perdue, les mises à jour en temps réel (logs, progression) ne sont plus reçues. L'acquisition peut continuer côté serveur, mais l'interface ne se met plus à jour. Dans ce cas, rafraîchir la page pour rétablir la connexion.

Étapes du cycle d'acquisition :

  1. L'utilisateur sélectionne le mode et la date/période, puis clique sur ▶ Lancer l'acquisition
  2. L'application vérifie que la session est valide et le token actif
  3. L'interface bascule : bouton Arrêter visible, badge "En cours…" affiché, carte Progression affichée
  4. La liste complète des trend collectors est récupérée depuis Desigo CC
  5. Pour chaque trend :
    • Récupération des données historiques sur la plage sélectionnée
    • Push SignalR → mise à jour des logs et de la progression
    • Sauvegarde ou traitement des données côté serveur
  6. Fin d'acquisition : carte Résultat affichée, interface restaurée

7. Gestion des Erreurs

7.1 Stratégies de Gestion

Type d'erreur Stratégie Action détaillée Impact utilisateur
Token expiré Alerte + blocage • Indicateur Token en rouge
• Message d'erreur dans les logs
• Acquisition bloquée jusqu'au refresh 		Cliquer sur 🔄 Refresh Token
Session perdue Réinitialisation • Dots de statut en rouge
• Message dans les logs
• Rechargement de la page requis 		Rafraîchir la page (F5)
Timeout API (trend) Skip + continue • Log d'avertissement
• Trend ignoré
• Passage au trend suivant 		Progression continue, trend omis
Trend invalide / 404 Skip + continue • Log d'erreur
• Trend ignoré
• Continuation de la boucle 		Progression continue
Perte SignalR Reconnexion auto • Tentative de reconnexion automatique
• Les logs ne s'affichent plus pendant la perte
• Acquisition continue côté serveur 		Rafraîchir la page si la reconnexion échoue
Erreur serveur (500) Log + stop • Message d'erreur détaillé dans les logs
• Arrêt de l'acquisition
• Interface restaurée 		Vérifier la configuration et réessayer
💡 Principe de conception : Comme pour le Trends Extractor automatique, l'application est conçue selon le principe "fail gracefully" : une erreur sur un trend individuel ne doit pas interrompre le traitement global. Seules les erreurs critiques (session perdue, erreur serveur) arrêtent l'acquisition.

7.2 Codes d'Erreur API

Code HTTP Signification Action Application Contexte typique
200 OK — Succès Traitement normal de la réponse Toutes les requêtes réussies
401 Non autorisé • Indicateur Token en rouge
• Message dans les logs
• Refresh Token requis 		Token expiré ou invalide
404 Non trouvé • Log d'avertissement
• Skip du trend
• Continuation 		Trend ou objet supprimé du système
500 Erreur serveur interne • Log d'erreur détaillé
• Arrêt de l'acquisition
• Interface restaurée 		Problème côté serveur Desigo CC
503 Service indisponible • Log d'erreur
• Arrêt de l'acquisition 		Serveur Desigo CC en maintenance
ℹ️ Timeout configuré : Toutes les requêtes HTTP ont un timeout de 180 secondes (3 minutes). Ce délai permet de gérer les requêtes de trends volumineux tout en évitant les blocages indéfinis.

8.4 FAQ

La barre de statut reste grise au chargement ?

Vérifier que le serveur Desigo CC est accessible depuis le réseau et que les paramètres de connexion dans appsettings.json (URL API, username, password) sont corrects. Consulter les logs dans la console pour identifier l'erreur précise.

L'acquisition se bloque sans message d'erreur ?

Vérifier la connexion SignalR dans les logs. Si aucun message n'apparaît depuis plusieurs secondes, la connexion SignalR est probablement perdue. Rafraîchir la page (F5) pour rétablir la connexion. L'acquisition peut avoir continué côté serveur.

Le token expire très rapidement ?

La durée du token est configurée côté serveur Desigo CC par l'administrateur système. L'application cliente ne peut pas modifier cette valeur. Utiliser le bouton 🔄 Refresh Token pour renouveler manuellement avant chaque extraction longue.

Peut-on lancer plusieurs acquisitions simultanément ?

Non. Une seule acquisition peut être active à la fois. Le bouton ▶ Lancer l'acquisition est remplacé par le bouton ⛔ Arrêter pendant une extraction. Il faut attendre la fin ou annuler l'acquisition en cours avant d'en lancer une nouvelle.

Les données extraites sont-elles sauvegardées automatiquement ?

Oui, les données sont traitées et sauvegardées côté serveur pendant l'acquisition. Le résultat final est accessible dans la carte Résultat à la fin de l'extraction.

Quelle est la différence avec le Trends Extractor automatique (v4.0) ?

Le Trends Extractor v4.0 est une application Windows Forms qui tourne en arrière-plan et extrait automatiquement les données toutes les X minutes. Le Manual Trend Extractor v.1 est une application web qui permet à un opérateur de déclencher manuellement une extraction sur une plage de dates précise. Les deux outils sont complémentaires.

✅ Conseil général : Avant toute extraction importante, vérifier que :
  • Les 3 indicateurs de statut (Session, Projet, Token) sont verts
  • Le token a une durée de validité suffisante pour l'extraction prévue
  • La connexion SignalR est établie (message 🔗 dans les logs)
  • La plage de dates sélectionnée est raisonnable (≤ 30 jours recommandé)
